Módulo 1

O que é Quality Assurance?

QA é muito mais do que encontrar bugs — é uma mentalidade de prevenção e garantia de qualidade em todo o ciclo de desenvolvimento de software.

🎯

Mindset de QA

O profissional de QA pensa no produto sob a perspectiva do usuário final, antecipando cenários de falha, bordas do sistema e comportamentos inesperados. A mentalidade é de prevenção proativa, não só de detecção reativa de defeitos. Um bom QA questiona requisitos, participa desde o início do projeto e atua como guardião da qualidade em toda a equipe.

🏋️ Exercícios Práticos
  • Escolha um app que você usa no dia a dia e liste 10 cenários de teste que um usuário comum poderia executar.
  • Para o mesmo app, identifique 5 cenários de borda (edge cases) que poderiam causar falhas.
  • Escreva um parágrafo descrevendo a diferença entre QA e QC (Quality Control) com exemplos.

✅

Verificação vs. Validação

Verificação responde: "Estamos construindo o produto corretamente?" — envolve revisões, inspeções e walkthroughs. Validação responde: "Estamos construindo o produto correto?" — envolve testar o software em condições reais para garantir que atende às necessidades do usuário. Ambas são complementares e essenciais.

🏋️ Exercícios Práticos
  • Dado um requisito de login, escreva 2 exemplos de atividade de verificação e 2 de validação.
  • Revise a documentação de um projeto open source e identifique lacunas nos requisitos.

Oráculos de Teste

Um oráculo de teste é o mecanismo utilizado para determinar se o software passou ou falhou em um teste. Pode ser a especificação do requisito, o comportamento esperado descrito em documentação, um sistema de referência, ou até o julgamento do próprio testador em testes exploratórios.

🏋️ Exercícios Práticos
  • Para uma calculadora simples, defina o oráculo de teste para as 4 operações básicas.
  • Identifique situações onde não existe um oráculo claro e como lidar com isso.
🧠

Priorização de Testes

Com tempo e recursos limitados, saber priorizar é essencial. Técnicas incluem: análise de risco (testar o que pode causar maior impacto), cobertura de requisitos críticos, histórico de defeitos, frequência de uso das funcionalidades e critérios de negócio. A pirâmide de testes é um guia fundamental para distribuição eficiente.

🏋️ Exercícios Práticos
  • Liste 10 funcionalidades de um e-commerce e priorize-as por risco e impacto de negócio.
  • Desenhe a pirâmide de testes para um projeto web que você conhece.
  • Crie uma matriz de risco simples para guiar seus testes.

Módulo 2

Abordagens de Teste

Existem três abordagens principais de teste baseadas no nível de conhecimento que o testador tem sobre o sistema.

⬜

Teste Caixa Branca (White Box)

O testador tem acesso ao código-fonte e testa a estrutura interna da aplicação. Foca em cobertura de código: cobertura de linhas, ramificações, condições e caminhos. É ideal para testes unitários e de integração, normalmente executados por desenvolvedores ou QAs com perfil técnico.

🏋️ Exercícios Práticos
  • Escreva testes unitários para uma função de validação de CPF em qualquer linguagem.
  • Use uma ferramenta de cobertura (ex: Istanbul/nyc para JS) e tente atingir 100% em uma função simples.
  • Analise um código com if/else e mapeie todos os caminhos possíveis de execução.
📚 Recursos
  • Unit Testing com Python — Udemy Pago
  • Jest Docs — Testes em JavaScript Gratuito
⬛

Teste Caixa Preta (Black Box)

O testador não conhece a implementação interna. Testa apenas o comportamento externo do sistema com base nos requisitos. É a abordagem mais comum em QA manual e automação funcional. Técnicas incluem: partição de equivalência, análise de valor limite, tabela de decisão e transição de estados.

🏋️ Exercícios Práticos
  • Para um campo de entrada de idade (0–120), aplique partição de equivalência e valor limite.
  • Crie uma tabela de decisão para um sistema de desconto com 3 condições de negócio.
  • Teste o site the-internet.herokuapp.com usando apenas caixa preta.
📚 Recursos
  • Testing do Zero ao Pleno — Udemy Pago
  • Black Box Testing Guide — Guru99 Gratuito
🔲

Teste Caixa Cinza (Gray Box)

Combinação das duas abordagens anteriores. O testador tem conhecimento parcial da implementação — acesso a banco de dados, logs, APIs internas — mas não ao código-fonte completo. Muito usado em testes de integração, segurança e quando o QA precisa validar dados diretamente na camada de persistência.

🏋️ Exercícios Práticos
  • Crie um teste de cadastro que valida os dados tanto pela UI quanto diretamente no banco de dados.
  • Use as DevTools do browser para inspecionar requisições e validar payloads de API durante um teste de UI.
Módulo 3

SDLC e Modelos de Entrega

O QA precisa entender os diferentes modelos de ciclo de vida de desenvolvimento para se adaptar à metodologia de cada projeto.

🌊

Modelo Waterfall

Processo sequencial com fases bem definidas: Requisitos → Design → Implementação → Verificação → Manutenção. Cada fase deve ser 100% concluída antes da próxima iniciar. O QA entra principalmente na fase de verificação. Menos comum hoje, mas ainda usado em projetos com requisitos fixos e contratos rígidos (ex: governo, contratos militares).

🏋️ Exercícios Práticos
  • Crie um plano de testes completo para um projeto hipotético seguindo a metodologia Waterfall.
  • Liste as vantagens e desvantagens do Waterfall para o trabalho de QA.
🔀

Modelo V (V-Model)

Extensão do Waterfall onde cada fase de desenvolvimento tem uma fase de teste correspondente. Os testes são planejados em paralelo ao desenvolvimento: testes unitários ↔ codificação, testes de integração ↔ design, testes de sistema ↔ especificação, testes de aceitação ↔ requisitos. Promove detecção precoce de defeitos.

🏋️ Exercícios Práticos
  • Desenhe o diagrama em "V" e posicione as atividades de QA em cada nível.
  • Para um projeto simples, defina os artefatos de teste para cada fase do V-Model.
🔁

Agile — Scrum e Kanban

No Scrum, o QA participa ativamente das sprints (2-4 semanas), escrevendo critérios de aceitação, automatizando testes e garantindo qualidade incremental. No Kanban, o foco é no fluxo contínuo. Ambos exigem que o QA seja multifuncional e colaborativo. O conceito de "Definition of Done" inclui obrigatoriamente critérios de qualidade.

🏋️ Exercícios Práticos
  • Participe de um projeto open source no GitHub e simule um ciclo de sprint com criação de user stories e critérios de aceitação.
  • Crie um board Kanban (no Trello ou Jira gratuito) para gerenciar suas atividades de estudo de QA.
  • Escreva 3 user stories com critérios de aceitação no formato BDD (Given/When/Then).
📚 Recursos
  • Scrum Guide (PT-BR disponível) — Scrum.org Gratuito
  • Agile Testing — Udemy Pago
  • Metodologias Ágeis — DIO Gratuito
🧩

SAFe e XP

SAFe (Scaled Agile Framework) é usado em grandes organizações para escalar o Agile. QAs trabalham tanto em times de feature quanto em papéis de arquitetura de qualidade. XP (Extreme Programming) enfatiza TDD, pair programming e integração contínua, colocando o QA como parte fundamental do processo de codificação.

🏋️ Exercícios Práticos
  • Pratique TDD: escreva o teste antes do código para uma função de calculadora de juros simples.
  • Pesquise como o SAFe organiza QA em grandes empresas e descreva o papel de "System Team".
Módulo 4

Testes Manuais

A base de todo bom testador. Testes manuais exigem atenção, criatividade e profundo conhecimento do sistema.

📄

Plano de Testes

Documento que descreve o escopo, abordagem, recursos e cronograma das atividades de teste. Um bom plano define: objetivos, critérios de entrada e saída, riscos identificados, ambiente de testes, estratégia de automação e alocação de responsabilidades. É o contrato entre o QA e os stakeholders do projeto.

🏋️ Exercícios Práticos
  • Crie um plano de testes para o site automationpractice.pl cobrindo funcionalidades de e-commerce.
  • Identifique 5 riscos técnicos e de negócio para um projeto de internet banking.

Casos e Cenários de Teste

Um caso de teste é um conjunto de condições para verificar um comportamento específico: pré-condições, passos, dados de entrada e resultado esperado. Um cenário de teste é uma situação de negócio de mais alto nível (ex: "usuário realiza compra com cartão de crédito"). Bons casos de teste são: rastreáveis, repetíveis, independentes e claros.

🏋️ Exercícios Práticos
  • Escreva 10 casos de teste para o formulário de login do site OrangeHRM.
  • Organize os casos por prioridade (alta, média, baixa) e justifique suas escolhas.
  • Documente um caso de teste usando o formato de tabela com ID, descrição, pré-condição, passos e resultado esperado.

HTML, CSS, JS e DevTools

Todo QA que trabalha com aplicações web precisa entender o básico de HTML/CSS para inspecionar elementos, JavaScript para entender comportamentos dinâmicos, e dominar as DevTools do browser para: verificar requisições de rede (Network tab), inspecionar o DOM, simular dispositivos, analisar performance e depurar erros no console.

🏋️ Exercícios Práticos
  • Abra o DevTools em qualquer site e intercepte uma requisição de login (aba Network). Analise os headers e o payload.
  • Use o console do browser para executar JavaScript e modificar o DOM de uma página.
  • Simule um dispositivo mobile e teste a responsividade de 3 sites diferentes.

Compatibilidade e Verificação

Testes de compatibilidade verificam se o software funciona corretamente em diferentes browsers (Chrome, Firefox, Safari, Edge), sistemas operacionais (Windows, macOS, Linux), resoluções e dispositivos. Ferramentas como BrowserStack permitem testar em ambientes reais remotos. Também inclui validar CSR vs SSR, PWAs, SPAs e responsividade.

🏋️ Exercícios Práticos
  • Teste o mesmo site em 3 browsers diferentes e documente as inconsistências visuais encontradas.
  • Crie uma matriz de compatibilidade para um projeto: liste OS × Browser × Dispositivo.

Módulo 5

Tipos de Teste Funcional

Cada tipo de teste tem um propósito específico dentro da estratégia de qualidade. Entender quando usar cada um é fundamental.

💨

Smoke Testing

Suite rápida de testes para verificar se o build básico está funcional antes de testes mais profundos. Cobre os fluxos mais críticos do sistema (ex: login, abertura da tela principal). Se o smoke falhar, o build é rejeitado sem gastar tempo em testes detalhados. Deve ser executado a cada novo deploy.

🏋️ Exercícios Práticos
  • Para um app de e-commerce, defina 5 casos de smoke test que cobrem os fluxos mais críticos.
  • Automatize esses 5 casos usando Cypress ou Playwright em menos de 2 minutos de execução.
🔁

Teste de Regressão

Garante que novas mudanças não quebraram funcionalidades existentes. É o candidato perfeito para automação, pois é repetitivo e precisa ser executado frequentemente. A suíte de regressão deve ser mantida e atualizada à medida que o sistema evolui. Ferramentas de CI/CD executam esses testes automaticamente a cada commit.

🏋️ Exercícios Práticos
  • Crie uma suíte de 10 casos de regressão para um módulo de cadastro de usuários.
  • Implemente esses testes com Playwright e configure para rodar no GitHub Actions.
🧩

Teste de Integração

Verifica a comunicação entre módulos ou serviços. Testa contratos de API, integrações com bancos de dados, filas de mensagens e serviços externos. É fundamental em arquiteturas de microsserviços para garantir que cada serviço se comunica corretamente com os demais.

🏋️ Exercícios Práticos
  • Use a API pública do reqres.in e crie testes de integração cobrindo CRUD completo com Postman.
  • Implemente os mesmos testes com REST Assured (Java) ou Supertest (Node.js).
🔍

Teste Exploratório

Abordagem simultânea de aprendizado, design e execução de testes. O testador usa seu conhecimento, intuição e criatividade para explorar o sistema sem scripts predefinidos. Muito eficaz para encontrar bugs que testes estruturados não capturam. Técnicas: charters de exploração, heurísticas (SFDIPOT, HICCUPPS) e sessões com time-box.

🏋️ Exercícios Práticos
  • Reserve 30 minutos para explorar o site demoqa.com e documente todos os bugs encontrados.
  • Crie um charter de exploração com missão, área e duração definidos.
📚 Recursos
  • Intro to Exploratory Testing — MoT Gratuito
👤

UAT (User Acceptance Testing)

Fase final antes do go-live onde usuários finais ou clientes validam se o sistema atende às necessidades de negócio. O QA apoia preparando ambiente, dados de teste e documentação. Resultados de UAT impactam diretamente a decisão de lançamento do produto.

🏋️ Exercícios Práticos
  • Crie um script de UAT para um módulo de emissão de nota fiscal, escrito em linguagem acessível para usuários não-técnicos.
  • Simule uma sessão de UAT com um colega que fará o papel de usuário final.
🧪

TDD (Test-Driven Development)

Metodologia onde os testes são escritos antes do código de produção. Ciclo: Red (teste falha) → Green (código mínimo para passar) → Refactor (melhorar sem quebrar). Melhora o design do código, reduz débito técnico e garante cobertura de testes nativa. Amplamente adotado em equipes ágeis maduras.

🏋️ Exercícios Práticos
  • Implemente uma função de FizzBuzz usando TDD com Jest ou Pytest.
  • Pratique TDD no site Coding Katas.

Módulo 6

Técnicas de Teste

Técnicas estruturadas para maximizar a cobertura e eficiência dos testes, minimizando o número de casos necessários.

⚖️

Partição de Equivalência

Divide os dados de entrada em grupos onde todos os valores se comportam da mesma forma. Ao testar um representante de cada grupo, reduzimos o número de casos sem perder cobertura. Ex: para "idade entre 18 e 65", existem 3 partições — menor que 18, entre 18-65 e maior que 65.

🏋️ Exercícios Práticos
  • Aplique partição de equivalência em um campo de CEP (8 dígitos) e liste todas as classes válidas e inválidas.
  • Crie casos de teste usando apenas um representante por partição para um sistema de classificação de IMC.
📏

Análise de Valor Limite

Complementa a partição de equivalência testando nos limites das partições, onde bugs são mais comuns. Para "1 a 100", testa-se: 0, 1, 2, 99, 100, 101. A lógica é que erros de programação (off-by-one) ocorrem nas bordas das condições.

🏋️ Exercícios Práticos
  • Para um desconto de 10% em compras entre R$100 e R$500, defina todos os valores limite a serem testados.
📊

Tabela de Decisão

Documenta sistematicamente combinações de condições e suas ações resultantes. Ideal para lógicas de negócio complexas com múltiplas condições interdependentes. Garante que todas as combinações relevantes sejam testadas, evitando casos omitidos.

🏋️ Exercícios Práticos
  • Crie uma tabela de decisão para um sistema de aprovação de crédito com: renda, score e histórico de inadimplência.
🔄

Transição de Estados

Modela o comportamento de sistemas que mudam de estado baseado em eventos. Muito útil para testar fluxos de pedido (Criado → Aprovado → Enviado → Entregue), autenticação e workflows. O diagrama de estados ajuda a identificar transições válidas e inválidas.

🏋️ Exercícios Práticos
  • Desenhe o diagrama de estados para um pedido de e-commerce e derive casos de teste para cada transição.
  • Identifique transições inválidas e crie casos de teste negativos para elas.
Módulo 7

Automação Frontend

Automatize testes de interface do usuário usando os principais frameworks do mercado para garantir qualidade de forma contínua e escalável.

🌲

Cypress

Framework moderno para testes end-to-end em JavaScript/TypeScript. Roda diretamente no browser, oferece debugging visual em tempo real, time-travel (snapshots de cada passo), interceptação de requisições e excelente DX (Developer Experience). Ideal para SPAs e aplicações React/Vue/Angular.

🏋️ Exercícios Práticos
  • Instale o Cypress e escreva testes para o fluxo de login/logout no site demoqa.com.
  • Implemente o Page Object Model (POM) nos seus testes Cypress.
  • Use cy.intercept() para interceptar e mockar uma chamada de API.
📚 Recursos
  • Documentação Oficial do Cypress Gratuito
  • Cypress — Web Automation Testing — Udemy Pago
  • Cypress Tutorial — TAU Applitools Gratuito
🎭

Playwright

Framework da Microsoft que suporta Chromium, Firefox e WebKit com a mesma API. Oferece paralelismo nativo, testes mobile com emulação, tracing detalhado, e suporte a múltiplas linguagens (JS, TS, Python, C#, Java). É considerado o mais completo e moderno framework de automação frontend em 2026.

🏋️ Exercícios Práticos
  • Configure um projeto Playwright com TypeScript e escreva testes cross-browser para um formulário de cadastro.
  • Use o Playwright Test Generator (codegen) para gerar testes automaticamente.
  • Configure execução paralela e gere relatório HTML dos testes.
📚 Recursos
  • Playwright — Documentação Oficial Gratuito
  • Playwright Automation Testing — Udemy Pago
🤖

Selenium WebDriver

O framework mais estabelecido do mercado, suportando múltiplas linguagens (Java, Python, C#, Ruby, JS) e browsers via WebDriver. Amplamente usado em projetos corporativos. Selenium 4 trouxe melhorias significativas: WebDriver BiDi, relative locators, e integração nativa com Chrome DevTools Protocol (CDP).

🏋️ Exercícios Práticos
  • Configure Selenium com Java + TestNG e escreva uma suíte básica de testes para the-internet.herokuapp.com.
  • Implemente o padrão Page Object Model com Selenium.
  • Integre Selenium com Allure Reports para geração de relatórios.
📚 Recursos
  • Selenium — Documentação Oficial Gratuito
  • Selenium + Java + TestNG — Udemy Pago
🐺

Outros Frameworks

O ecossistema é vasto: Robot Framework (keyword-driven, Python); WebdriverIO (JS/TS, multi-runner); Nightwatch.js (JS simplificado); Puppeteer (headless Chrome da Google); Jest + Testing Library (testes unitários e de componentes React); QA Wolf (geração automática com IA).

Módulo 8

Automação Backend e API

Testar APIs é essencial na era dos microsserviços. APIs bem testadas garantem contratos sólidos entre front e back-end.

📮

Postman / Newman

Postman é a ferramenta mais popular para teste manual e automatizado de APIs REST. Permite criar coleções, environments, scripts de pré/pós-requisição e testes com assertions em JavaScript. Newman é o CLI do Postman que permite executar coleções no terminal e em pipelines de CI/CD.

🏋️ Exercícios Práticos
  • Crie uma coleção Postman para a API reqres.in cobrindo todos os endpoints (CRUD de usuários).
  • Adicione assertions de status code, tempo de resposta e schema validation em cada request.
  • Execute a coleção via Newman e integre ao GitHub Actions.
📚 Recursos
  • Postman Learning Center Gratuito
  • Postman Complete Guide — Udemy Pago
☕

REST Assured

Biblioteca Java para testes de APIs REST/HTTP. Usa uma sintaxe fluente (Given/When/Then) que facilita a leitura e escrita dos testes. Muito popular em projetos Java com Maven/Gradle. Suporta autenticação OAuth, parsing de JSON/XML, e validação de schemas.

🏋️ Exercícios Práticos
  • Configure um projeto Maven com REST Assured + TestNG e teste a API do JSONPlaceholder.
  • Implemente validação de schema JSON com JSONSchemaValidator.
  • Crie testes para endpoints autenticados com JWT.
📚 Recursos
  • REST Assured Docs Gratuito
  • REST API Testing com REST Assured — Udemy Pago
🥋

Karate Framework

Framework open-source que combina automação de APIs, testes de performance (Gatling integrado) e até UI em um único DSL baseado em Gherkin. Não requer conhecimento de Java para escrever testes. Excelente para equipes que precisam de uma solução unificada para múltiplos tipos de teste.

🏋️ Exercícios Práticos
  • Configure um projeto Karate e escreva testes para a API pública do GitHub.
  • Use o recurso de parallel execution do Karate para otimizar o tempo de execução.
📚 Recursos
  • Karate — GitHub Oficial Gratuito
🧼

SoapUI / Web Services

Ferramenta especializada em testes de Web Services SOAP e REST. Muito usada em ambientes corporativos e projetos que ainda utilizam SOAP/WSDL. A versão ReadyAPI (paga) adiciona funcionalidades de performance e segurança. Para APIs modernas, Postman e REST Assured são mais comuns.

🏋️ Exercícios Práticos
  • Use SoapUI open source para testar um serviço SOAP público (ex: calculator WSDL).
Módulo 9

Automação Mobile

Testes automatizados para aplicações Android e iOS — uma habilidade cada vez mais valorizada no mercado.

📱

Appium

Framework open-source multiplataforma para automação de apps Android e iOS. Usa o protocolo WebDriver e permite escrever testes em diversas linguagens. Suporta apps nativos, híbridos e web mobile. A versão Appium 2.x trouxe arquitetura modular com drivers independentes.

🏋️ Exercícios Práticos
  • Configure o ambiente Appium com Android Emulator e automatize o app de calculadora nativa do Android.
  • Escreva testes para gestos (swipe, pinch, tap) usando Appium.
  • Integre Appium com BrowserStack para testes em devices reais na nuvem.

Espresso (Android)

Framework oficial do Google para testes de UI no Android. Roda diretamente no dispositivo/emulador, oferece sincronização automática com threads da UI e é muito mais rápido que Appium para testes unitários de UI. Ideal para desenvolvedores Android que querem implementar testes de interface.

🏋️ Exercícios Práticos
  • Crie um projeto Android Studio e implemente testes Espresso para uma tela de login.

XCUITest (iOS) e Detox

XCUITest é o framework oficial da Apple para automação de UI no iOS/macOS, integrado ao Xcode. Detox é focado em apps React Native, oferecendo testes gray-box que sincronizam automaticamente com o estado da aplicação, reduzindo flakiness em apps híbridos.

🏋️ Exercícios Práticos
  • Se tiver acesso a um Mac, configure o Detox em um projeto React Native de exemplo.
  • Pesquise as diferenças entre Appium, Detox e XCUITest para apps React Native e documente sua análise.
Módulo 10

Performance e Testes de Carga

Garantir que o sistema aguenta a demanda esperada — e a inesperada — é responsabilidade do QA moderno.

📊

Tipos: Load, Stress, Spike

Load Testing: verifica comportamento sob carga normal esperada. Stress Testing: empurra o sistema além do limite para encontrar o ponto de ruptura. Spike Testing: simula picos repentinos de acesso. Soak Testing: carga constante por longo período para detectar memory leaks. Volume Testing: grandes volumes de dados.

🏋️ Exercícios Práticos
  • Crie um teste de carga com K6 para uma API pública, simulando 100 usuários simultâneos por 1 minuto.
  • Analise os resultados e identifique gargalos (p95, p99 latency, error rate).
  • Documente a diferença de comportamento entre Load e Stress test no mesmo endpoint.
📚 Recursos
  • K6 — Documentação Oficial (Grafana) Gratuito
  • Apache JMeter — User Manual Gratuito
  • JMeter Performance Testing — Udemy Pago
🚀

K6, Gatling e Artillery

K6 (Grafana): moderno, scripts em JS/TS, integra com Grafana/Prometheus. Gatling: alto desempenho em JVM, relatórios HTML bonitos, integra com Karate. Artillery: baseado em Node.js, fácil de usar, suporte a WebSockets. Locust: Python, intuitivo para testes distribuídos. JMeter: mais completo, GUI amigável, protocolo HTTP/JDBC/JMS.

K6 JMeter Gatling Artillery Locust Vegeta Lighthouse
🏋️ Exercícios Práticos
  • Use o Lighthouse CLI para analisar a performance de 3 sites e compare os scores.
  • Configure Grafana + InfluxDB + K6 para visualizar resultados em tempo real.
Módulo 11

Testes de Segurança

QA moderno precisa entender os fundamentos de segurança para identificar vulnerabilidades antes que sejam exploradas.

🔐

OWASP Top 10

As 10 vulnerabilidades mais críticas em aplicações web, mantidas pela OWASP Foundation. Incluem: Broken Access Control, Criptografia falha, Injection (SQL, XSS, etc.), Design Inseguro, Configuração incorreta de segurança, Componentes vulneráveis, Falhas de autenticação, Integridade de software e dados, Logging insuficiente e SSRF.

🏋️ Exercícios Práticos
  • Teste as vulnerabilidades OWASP no app vulnerável WebGoat (ambiente controlado).
  • Use o OWASP ZAP para fazer um scan automatizado em uma aplicação de teste.
  • Documente como testar SQL Injection de forma ética em ambientes de desenvolvimento.
📚 Recursos
  • OWASP Top 10 — Oficial Gratuito
  • Web Security Testing — Udemy Pago
  • Web Security Academy — PortSwigger Gratuito
🔑

Autenticação, Autorização e Secrets

Testes de autenticação verificam se o sistema valida corretamente as credenciais e tokens (JWT, OAuth, API Keys). Testes de autorização garantem que cada usuário acessa apenas o que tem permissão (RBAC). Gerenciamento de secrets é crítico: chaves de API, senhas e certificados nunca devem estar expostos no código.

🏋️ Exercícios Práticos
  • Teste cenários de IDOR (Insecure Direct Object Reference) em uma API de teste.
  • Use o jwt.io para decodificar e analisar tokens JWT e identificar informações sensíveis expostas.
  • Configure o git-secrets ou truffleHog para detectar secrets no código.
📧

Teste de Email

Verificar o comportamento de emails enviados pela aplicação é parte da qualidade. Ferramentas como Mailinator oferecem caixas de entrada temporárias públicas para testes de cadastro/confirmação. Gmail Tester permite integrar testes de email com contas Gmail via API para validações automatizadas.

🏋️ Exercícios Práticos
  • Use o Mailinator para testar o fluxo completo de registro → confirmação de email em uma aplicação.
  • Valide conteúdo do email, links e tempo de entrega como parte do seu teste.
Módulo 12

Testes de Acessibilidade

Garantir que aplicações sejam utilizáveis por todos — incluindo pessoas com deficiência — é uma responsabilidade ética e legal.

♿

Diretrizes WCAG

As Web Content Accessibility Guidelines (WCAG) definem padrões de acessibilidade organizados em 4 princípios: Perceptível (alternativas de texto, legendas), Operável (navegação por teclado), Compreensível (linguagem clara, erros identificáveis) e Robusto (compatível com tecnologias assistivas). WCAG 2.1 nível AA é o padrão mais exigido.

🏋️ Exercícios Práticos
  • Navegue em um site usando apenas o teclado (Tab, Enter, Esc) e documente problemas de foco encontrados.
  • Use um leitor de telas (NVDA gratuito para Windows) para navegar em um site e identificar barreiras.
  • Verifique o contraste de cores de um site usando o WebAIM Contrast Checker.
📚 Recursos
  • WCAG 2.1 Quick Reference — W3C Gratuito
  • Website Accessibility Testing — Udemy Pago
🔦

Axe, WAVE e Lighthouse

Axe (Deque Systems): extensão de browser e biblioteca JavaScript que detecta automaticamente violações WCAG. WAVE: extensão visual que mostra erros diretamente na página. Lighthouse: auditoria integrada ao Chrome DevTools que inclui score de acessibilidade. Nenhuma ferramenta detecta 100% dos problemas — testes manuais são sempre necessários.

🏋️ Exercícios Práticos
  • Instale a extensão Axe DevTools e faça uma auditoria em 3 sites populares brasileiros.
  • Integre o axe-core com Playwright para testes automatizados de acessibilidade.
  • Gere um relatório Lighthouse para um site e corrija pelo menos 3 problemas de acessibilidade encontrados.
📚 Recursos
  • Axe DevTools — Deque Gratuito
  • WAVE — WebAIM Gratuito
Módulo 13

Controle de Versão — Git

Git é uma habilidade fundamental para todo QA moderno. Seu código de teste precisa ser versionado, revisado e colaborado como qualquer outro código.

🌿

Git — Fundamentos

Comandos essenciais: clone, init, add, commit, push, pull, fetch, merge, rebase, branch, checkout, stash, log, diff. QAs precisam entender branching strategies (GitFlow, trunk-based), pull requests e code review. Seus testes automatizados devem viver no mesmo repositório do código ou em um repo dedicado bem estruturado.

🏋️ Exercícios Práticos
  • Crie um repositório no GitHub para seu projeto de automação e faça commit de testes com mensagens convencionais (Conventional Commits).
  • Pratique resolução de conflitos de merge em um repositório de teste.
  • Configure um .gitignore adequado para projetos de automação (excluindo node_modules, reports, screenshots, etc.).
📚 Recursos
  • Learn Git Branching — Interativo Gratuito
  • Git e GitHub — Alura Pago

🐙

GitHub, GitLab e Bitbucket

Plataformas de hospedagem e colaboração de código. Além de repositórios, oferecem: Issues (gestão de bugs), Pull Requests (code review), Actions/Pipelines (CI/CD), Packages (registry), Pages (documentação). QAs usam essas plataformas para gerenciar suítes de teste, automatizar execuções e colaborar com o time de desenvolvimento.

🏋️ Exercícios Práticos
  • Configure um repositório público no GitHub com um projeto de automação e adicione um README detalhado.
  • Use GitHub Issues para registrar e rastrear bugs encontrados durante testes exploratórios.
Módulo 14

CI/CD — Integração e Entrega Contínua

Integrar testes automatizados ao pipeline de CI/CD é o que transforma QA em um guardião contínuo da qualidade.

⚙️

GitHub Actions

Plataforma de CI/CD nativa do GitHub que permite criar workflows YAML para automatizar testes a cada push/PR. É gratuita para projetos públicos e tem generoso plano gratuito para privados. Suporta runners managed pelo GitHub ou self-hosted, matrix builds para testar em múltiplos ambientes simultaneamente.

🏋️ Exercícios Práticos
  • Crie um workflow GitHub Actions que executa seus testes Playwright automaticamente a cada PR.
  • Configure matrix strategy para rodar testes em Chrome, Firefox e WebKit simultaneamente.
  • Publique o relatório HTML do Playwright como GitHub Pages do repositório.
📚 Recursos
  • GitHub Actions — Documentação Oficial Gratuito
  • GitHub Actions — Udemy Pago
🏗️

Jenkins, GitLab CI e Outros

Jenkins: open-source, altamente configurável, vasta biblioteca de plugins, deploy on-premise. GitLab CI: integrado ao GitLab, simples e poderoso. Azure DevOps: solução completa da Microsoft (Repos + Pipelines + Boards). CircleCI e Travis CI: populares em projetos open-source.

Jenkins GitLab CI Azure DevOps CircleCI TeamCity Drone CI
🏋️ Exercícios Práticos
  • Configure um pipeline Jenkins local (via Docker) que executa testes Maven/JUnit.

🕶️

Headless Testing

Execução de testes de browser sem interface gráfica — essencial em ambientes de CI/CD onde não há display disponível. Frameworks como Playwright, Cypress e Puppeteer suportam modo headless nativamente. O Headless Chrome e Firefox rodam com flag --headless. Mais rápido que modo GUI e ideal para pipelines de automação.

🏋️ Exercícios Práticos
  • Configure seus testes Playwright para rodar em modo headless via linha de comando.
  • Execute os testes dentro de um container Docker sem interface gráfica.
Módulo 15

Monitoramento, Logs e Relatórios

QA não termina no deploy. Monitorar a saúde da aplicação em produção e reagir rapidamente a problemas é parte do ciclo de qualidade.

📈

Grafana, Datadog e New Relic

Grafana: dashboards open-source para visualização de métricas (CPU, memória, latência, error rate). Datadog: plataforma APM completa com rastreamento distribuído, logs e alertas inteligentes. New Relic: similar ao Datadog, foco em observabilidade full-stack. QAs usam essas ferramentas para detectar regressões de performance em produção.

🏋️ Exercícios Práticos
  • Configure Grafana + Prometheus localmente via Docker Compose e crie um dashboard básico.
  • Crie alertas para quando a taxa de erro de uma API ultrapasse 1%.
  • Configure Sentry em uma aplicação de exemplo para capturar erros automaticamente.
📚 Recursos
  • Grafana Tutorials — Oficial Gratuito
  • Sentry — Documentação Gratuito
📋

Allure e Relatórios de Teste

Allure Framework: framework open-source para geração de relatórios de teste bonitos e informativos com gráficos, histórico de execuções, screenshots e logs de falha. Integra com TestNG, JUnit, Pytest, Cucumber e Playwright. jUnit XML: formato padrão para integração com ferramentas de CI.

🏋️ Exercícios Práticos
  • Integre Allure ao seu projeto de automação (Selenium+TestNG ou Pytest) e gere um relatório completo.
  • Configure Allure para incluir screenshots dos testes que falharam.
  • Publique o relatório Allure no GitHub Pages do seu repositório.
📚 Recursos
  • Allure Framework — Docs Oficial Gratuito
🔍

Kibana, Runscope e PagerDuty

Kibana (ELK Stack): visualização e análise de logs com Elasticsearch. Fundamental para investigar falhas. Runscope: monitoramento contínuo de APIs em produção. PagerDuty: gestão de alertas e on-call, notifica a equipe quando incidentes críticos são detectados. QAs definem alertas e SLOs (Service Level Objectives).

🏋️ Exercícios Práticos
  • Configure a stack ELK (Elasticsearch + Logstash + Kibana) via Docker e analise logs de uma aplicação.
  • Crie uma query KQL no Kibana para filtrar logs de erros de uma aplicação específica.
