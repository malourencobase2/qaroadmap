Módulo 1

O que é Quality Assurance?

QA é muito mais do que encontrar bugs — é uma mentalidade de prevenção e garantia de qualidade em todo o ciclo de desenvolvimento de software.

🎯

Mindset de QA

O profissional de QA pensa no produto sob a perspectiva do usuário final, antecipando cenários de falha, bordas do sistema e comportamentos inesperados. A mentalidade é de prevenção proativa, não só de detecção reativa de defeitos. Um bom QA questiona requisitos, participa desde o início do projeto e atua como guardião da qualidade em toda a equipe.

🏋️ Exercícios Práticos
  • Escolha um app que você usa no dia a dia e liste 10 cenários de teste que um usuário comum poderia executar.
  • Para o mesmo app, identifique 5 cenários de borda (edge cases) que poderiam causar falhas.
  • Escreva um parágrafo descrevendo a diferença entre QA e QC (Quality Control) com exemplos.

Verificação vs. Validação

Verificação responde: "Estamos construindo o produto corretamente?" — envolve revisões, inspeções e walkthroughs. Validação responde: "Estamos construindo o produto correto?" — envolve testar o software em condições reais. Ambas são complementares e essenciais.

🏋️ Exercícios Práticos
  • Dado um requisito de login, escreva 2 exemplos de atividade de verificação e 2 de validação.
  • Revise a documentação de um projeto open source e identifique lacunas nos requisitos.

Oráculos de Teste

Um oráculo de teste é o mecanismo utilizado para determinar se o software passou ou falhou em um teste. Pode ser a especificação do requisito, o comportamento esperado descrito em documentação, um sistema de referência, ou até o julgamento do próprio testador em testes exploratórios.

🏋️ Exercícios Práticos
  • Para uma calculadora simples, defina o oráculo de teste para as 4 operações básicas.
  • Identifique situações onde não existe um oráculo claro e como lidar com isso.
🧠

Priorização de Testes

Com tempo e recursos limitados, saber priorizar é essencial. Técnicas incluem: análise de risco, cobertura de requisitos críticos, histórico de defeitos, frequência de uso e critérios de negócio. A pirâmide de testes é um guia fundamental para distribuição eficiente.

🏋️ Exercícios Práticos
  • Liste 10 funcionalidades de um e-commerce e priorize-as por risco e impacto de negócio.
  • Desenhe a pirâmide de testes para um projeto web que você conhece.
  • Crie uma matriz de risco simples para guiar seus testes.

Laboratório Prático — Fundamentos de QA

Exercícios guiados para fixar os conceitos essenciais de qualidade de software

1
Mapeando o mindset de QA em um app real

Escolha qualquer app instalado no seu celular e analise-o com o olhar crítico de um QA. O objetivo é treinar a percepção de cenários de falha antes de qualquer teste formal.

  • 1. Abra o app e navegue por 5 minutos como um usuário comum — anote o fluxo principal que você seguiu.
  • 2. Repita a navegação pensando: "O que poderia dar errado aqui?" — liste pelo menos 10 perguntas do tipo "E se...?"
  • 3. Para cada pergunta, classifique: é um risco de dados inválidos, performance, segurança ou usabilidade?
  • 4. Documente numa planilha: Funcionalidade | Cenário de Risco | Tipo | Prioridade (Alta/Média/Baixa).
💡 Dica de início: Comece pelo fluxo de login/cadastro — é onde a maioria dos bugs críticos se esconde. Pense em: senha errada, campo vazio, caracteres especiais, conexão lenta.
2
Verificação vs. Validação na prática

Use o repositório público public-apis para distinguir as duas atividades no mundo real.

  • 1. Acesse o repositório e leia o README por 10 minutos.
  • 2. Liste 3 atividades de verificação que um QA poderia fazer (ex: revisar se a documentação descreve corretamente os endpoints).
  • 3. Liste 3 atividades de validação (ex: chamar um endpoint e verificar se a resposta atende às necessidades do usuário).
  • 4. Escreva um parágrafo com suas conclusões: por que as duas atividades são complementares?
💡 Dica de início: Verificação é sobre processo e documentação — não precisa rodar código. Validação exige executar algo e comparar com a expectativa do usuário final.
3
Pirâmide de testes: diagnóstico de um projeto real

Analise a distribuição de testes de um projeto open source e compare com a pirâmide ideal.

  • 1. Acesse github.com/gothinkster/realworld e explore a pasta de testes de qualquer implementação.
  • 2. Conte quantos arquivos de teste existem por camada: unitários, integração e E2E.
  • 3. Desenhe a pirâmide atual e a ideal — compare as diferenças.
  • 4. Escreva 3 recomendações concretas para melhorar a estratégia de testes.
💡 Dica de início: A pirâmide ideal tem muitos testes unitários na base e poucos E2E no topo. Se há mais E2E que unitários, é sinal de alerta — testes lentos e frágeis dominam.
Módulo 2

Abordagens de Teste

Existem três abordagens principais de teste baseadas no nível de conhecimento que o testador tem sobre o sistema.

⬜

Teste Caixa Branca (White Box)

O testador tem acesso ao código-fonte e testa a estrutura interna da aplicação. Foca em cobertura de código: linhas, ramificações, condições e caminhos. Ideal para testes unitários e de integração, normalmente executados por desenvolvedores ou QAs com perfil técnico.

🏋️ Exercícios Práticos
  • Escreva testes unitários para uma função de validação de CPF em qualquer linguagem.
  • Use Istanbul/nyc para JS e tente atingir 100% de cobertura em uma função simples.
  • Analise um código com if/else e mapeie todos os caminhos possíveis de execução.
📚 Cursos e Recursos
  • Jest Docs — Testes em JavaScript Gratuito
  • Unit Testing com Python — Udemy Pago
⬛

Teste Caixa Preta (Black Box)

O testador não conhece a implementação interna. Testa apenas o comportamento externo com base nos requisitos. É a abordagem mais comum em QA manual e automação funcional. Técnicas incluem: partição de equivalência, análise de valor limite, tabela de decisão e transição de estados.

🏋️ Exercícios Práticos
  • Para um campo de idade (0–120), aplique partição de equivalência e valor limite.
  • Crie uma tabela de decisão para um sistema de desconto com 3 condições de negócio.
  • Teste o site the-internet.herokuapp.com usando apenas caixa preta.
📚 Cursos e Recursos
  • Black Box Testing Guide — Guru99 Gratuito
  • Testing do Zero ao Pleno — Udemy Pago
🔲

Teste Caixa Cinza (Gray Box)

Combinação das duas abordagens. O testador tem conhecimento parcial da implementação — acesso a banco de dados, logs, APIs internas — mas não ao código-fonte completo. Muito usado em testes de integração, segurança e quando o QA precisa validar dados diretamente na camada de persistência.

🏋️ Exercícios Práticos
  • Crie um teste de cadastro que valida os dados tanto pela UI quanto diretamente no banco de dados.
  • Use as DevTools do browser para inspecionar requisições e validar payloads de API durante um teste de UI.

Laboratório Prático — Abordagens de Teste

Caixa branca com Jest, caixa preta no The Internet e caixa cinza com Postman

1
Caixa Branca: cobertura 100% de branches com Jest

Escreva testes de caixa branca para uma função de desconto buscando 100% de cobertura de branches. Antes de qualquer teste, desenhe o fluxograma da função.

  • 1. Setup: npm init -y && npm i -D jest. Crie discount.js com a função abaixo.
  • 2. Analise todos os caminhos (cada if/else é pelo menos 2 caminhos). Desenhe o fluxograma no papel.
  • 3. Escreva um teste Jest para cada caminho — mínimo 5 casos diferentes.
  • 4. Rode npx jest --coverage. Objetivo: 100% em Branches. Adicione casos até chegar lá.
// discount.js function calcDiscount(price, userType, coupon) { if (price <= 0) throw new Error('Preco invalido'); let discount = 0; if (userType === 'vip') discount += 0.20; else if (userType === 'member') discount += 0.10; if (coupon === 'EXTRA10') discount += 0.10; if (discount > 0.30) discount = 0.30; return parseFloat((price * (1 - discount)).toFixed(2)); } module.exports = calcDiscount;
💡 Dica de início: No Jest, use expect(() => fn()).toThrow() para testar exceções. Cada losango no fluxograma = pelo menos 2 casos de teste (verdadeiro e falso).
2
Caixa Preta: partição de equivalência no The Internet

Aplique técnicas de caixa preta no the-internet.herokuapp.com/login sem nenhum conhecimento da implementação.

  • 1. Defina as partições: credenciais válidas, usuário inválido, senha inválida, ambos inválidos, campos vazios, espaços em branco.
  • 2. Escreva um caso de teste por partição em tabela: ID | Entrada | Resultado Esperado | Resultado Obtido | Status.
  • 3. Execute todos os casos manualmente e preencha os resultados obtidos.
  • 4. Identifique resultados surpreendentes — cada um é candidato a bug ou documentação insuficiente.
💡 Dica de início: Credenciais válidas: tomsmith / SuperSecretPassword! — não tente força bruta, o site é de prática controlado.
3
Caixa Cinza: validação de schema com Postman

Teste a API JSONPlaceholder validando não só o comportamento externo mas a estrutura interna da resposta.

  • 1. No Postman, faça GET /users/1 no JSONPlaceholder.
  • 2. Na aba Tests, use o snippet "Response body: JSON schema validation" para gerar o schema automaticamente.
  • 3. Edite o schema: adicione "required": ["id","name","email","address"] e tipos estritos para cada campo.
  • 4. Documente quais informações do "interior" você precisou conhecer para escrever as validações.
💡 Dica de início: Clique no botão Snippets no lado direito da aba Tests — há dezenas de assertions prontas para copiar e adaptar. O schema JSON garante que mudanças no backend não passem despercebidas.
Módulo 3

SDLC e Modelos de Entrega

O QA precisa entender os diferentes modelos de ciclo de vida de desenvolvimento para se adaptar à metodologia de cada projeto.

🌊

Modelo Waterfall

Processo sequencial com fases bem definidas: Requisitos → Design → Implementação → Verificação → Manutenção. O QA entra principalmente na fase de verificação. Menos comum hoje, mas ainda usado em projetos com requisitos fixos e contratos rígidos (ex: governo, contratos militares).

🏋️ Exercícios Práticos
  • Crie um plano de testes completo para um projeto hipotético seguindo a metodologia Waterfall.
  • Liste as vantagens e desvantagens do Waterfall para o trabalho de QA.
🔀

Modelo V (V-Model)

Extensão do Waterfall onde cada fase de desenvolvimento tem uma fase de teste correspondente: testes unitários ↔ codificação, integração ↔ design, sistema ↔ especificação, aceitação ↔ requisitos. Promove detecção precoce de defeitos.

🏋️ Exercícios Práticos
  • Desenhe o diagrama em "V" e posicione as atividades de QA em cada nível.
  • Para um projeto simples, defina os artefatos de teste para cada fase do V-Model.
🔁

Agile — Scrum e Kanban

No Scrum, o QA participa ativamente das sprints, escrevendo critérios de aceitação e automatizando testes. No Kanban, o foco é no fluxo contínuo. O conceito de "Definition of Done" inclui obrigatoriamente critérios de qualidade.

🏋️ Exercícios Práticos
  • Crie um board no Trello com colunas de sprint e pratique escrita de critérios de aceitação.
  • Escreva 3 user stories com critérios no formato BDD (Given/When/Then).

SAFe e XP

SAFe é usado em grandes organizações para escalar o Agile. XP (Extreme Programming) enfatiza TDD, pair programming e integração contínua, colocando o QA como parte fundamental do processo de codificação.

🏋️ Exercícios Práticos
  • Pratique TDD: escreva o teste antes do código para uma função de calculadora de juros simples.
  • Pesquise como o SAFe organiza QA em grandes empresas e descreva o papel de "System Team".

Laboratório Prático — SDLC e Modelos de Entrega

Simule um sprint Scrum e pratique TDD com o ciclo Red-Green-Refactor

1
Simulando um sprint Scrum como QA

Monte um mini-sprint de 1 hora simulando seu papel como QA em uma equipe Scrum real usando o Trello gratuito.

  • 1. Crie um board com colunas: Backlog | Sprint | Em Desenvolvimento | Em QA | Concluído.
  • 2. Crie 5 user stories para um app de tarefas no formato: "Como [usuário], quero [ação] para [benefício]".
  • 3. Para cada story, adicione um checklist de Critérios de Aceitação — mínimo 3 por story, todos testáveis e objetivos.
  • 4. Mova 2 stories para "Em QA", escreva os casos de teste no card e documente o resultado de cada um.
💡 Dica de início: Critérios ruins: "deve funcionar bem". Critérios bons: "deve exibir mensagem de erro em menos de 2s quando o campo e-mail estiver vazio e o usuário clicar em Salvar".
2
TDD: carrinho de compras com ciclo Red-Green-Refactor

Implemente um carrinho de compras usando TDD estrito — nenhuma linha de produção antes de um teste falhando.

  • 1. Setup: mkdir tdd-cart && cd tdd-cart && npm init -y && npm i -D jest. Crie cart.test.js (testes) e cart.js (vazio por enquanto).
  • 2. Escreva o primeiro teste que vai falhar (Red): expect(addItem({}, "Maçã", 2.5)).toEqual({items:[{name:"Maçã",price:2.5}],total:2.5}). Rode: npx jest --watch.
  • 3. Escreva o mínimo de código para passar (Green). Depois refatore sem quebrar. Repita para: remover item, calcular desconto, limpar carrinho.
  • 4. Ao final, rode npx jest --coverage — com TDD a cobertura tende a ser naturalmente alta.
💡 Dica de início: A tentação é escrever código completo de uma vez. Resista! Escreva literalmente o mínimo. Se o teste espera um objeto, sua primeira implementação pode retornar o objeto hardcoded — só depois generalize.

Plano de testes para o OrangeHRM

Módulo 5

Tipos de Teste Funcional

Cada tipo de teste tem um propósito específico dentro da estratégia de qualidade. Entender quando usar cada um é fundamental.

💨

Smoke Testing

Suite rápida de testes para verificar se o build básico está funcional antes de testes mais profundos. Cobre os fluxos mais críticos (ex: login, tela principal). Se o smoke falhar, o build é rejeitado. Deve ser executado a cada novo deploy.

🏋️ Exercícios Práticos
  • Para um app de e-commerce, defina 5 casos de smoke test que cobrem os fluxos mais críticos.
  • Automatize esses 5 casos usando Cypress ou Playwright em menos de 2 minutos de execução.
🔁

Teste de Regressão

Garante que novas mudanças não quebraram funcionalidades existentes. É o candidato perfeito para automação, pois é repetitivo e frequente. A suíte de regressão deve ser mantida e atualizada à medida que o sistema evolui.

🏋️ Exercícios Práticos
  • Crie uma suíte de 10 casos de regressão para um módulo de cadastro de usuários.
  • Implemente esses testes com Playwright e configure para rodar no GitHub Actions.
🧩

Teste de Integração

Verifica a comunicação entre módulos ou serviços. Testa contratos de API, integrações com bancos de dados, filas de mensagens e serviços externos. Fundamental em arquiteturas de microsserviços.

🏋️ Exercícios Práticos
  • Use a API pública reqres.in e crie testes de integração cobrindo CRUD completo com Postman.
  • Implemente os mesmos testes com REST Assured (Java) ou Supertest (Node.js).
🔍

Teste Exploratório

Abordagem simultânea de aprendizado, design e execução de testes. O testador usa conhecimento, intuição e criatividade sem scripts predefinidos. Muito eficaz para encontrar bugs que testes estruturados não capturam. Técnicas: charters, heurísticas (SFDIPOT, HICCUPPS) e sessões com time-box.

🏋️ Exercícios Práticos
  • Reserve 30 minutos para explorar o site demoqa.com e documente todos os bugs encontrados.
  • Crie um charter de exploração com missão, área e duração definidos.
📚 Cursos e Recursos
  • Intro to Exploratory Testing — MoT Gratuito
👤

UAT (User Acceptance Testing)

Fase final antes do go-live onde usuários finais validam se o sistema atende às necessidades de negócio. O QA apoia preparando ambiente, dados de teste e documentação. Resultados de UAT impactam diretamente a decisão de lançamento.

🏋️ Exercícios Práticos
  • Crie um script de UAT para um módulo de nota fiscal em linguagem acessível para usuários não-técnicos.
  • Simule uma sessão de UAT com um colega que fará o papel de usuário final.

Módulo 6

Técnicas de Teste

Técnicas estruturadas para maximizar a cobertura e eficiência dos testes, minimizando o número de casos necessários.

⚖️

Partição de Equivalência

Divide os dados de entrada em grupos onde todos os valores se comportam da mesma forma. Ao testar um representante de cada grupo, reduzimos o número de casos sem perder cobertura. Ex: para "idade entre 18 e 65", existem 3 partições: menor que 18, entre 18-65 e maior que 65.

🏋️ Exercícios Práticos
  • Aplique partição de equivalência em um campo de CEP (8 dígitos) e liste todas as classes válidas e inválidas.
  • Crie casos de teste usando apenas um representante por partição para um sistema de classificação de IMC.
📏

Análise de Valor Limite

Complementa a partição de equivalência testando nos limites das partições, onde bugs são mais comuns. Para "1 a 100", testa-se: 0, 1, 2, 99, 100, 101. Erros de programação (off-by-one) ocorrem nas bordas das condições.

🏋️ Exercícios Práticos
  • Para um desconto de 10% em compras entre R$100 e R$500, defina todos os valores limite a serem testados.
📊

Tabela de Decisão

Documenta sistematicamente combinações de condições e suas ações resultantes. Ideal para lógicas de negócio complexas com múltiplas condições interdependentes. Garante que todas as combinações relevantes sejam testadas.

🏋️ Exercícios Práticos
  • Crie uma tabela de decisão para um sistema de aprovação de crédito com: renda, score e histórico de inadimplência.
🔄

Transição de Estados

Modela o comportamento de sistemas que mudam de estado baseado em eventos. Muito útil para testar fluxos de pedido (Criado → Aprovado → Enviado → Entregue), autenticação e workflows.

🏋️ Exercícios Práticos
  • Desenhe o diagrama de estados para um pedido de e-commerce e derive casos de teste para cada transição.
  • Identifique transições inválidas e crie casos de teste negativos para elas.

Partição + Valor Limite: validação de senha

Regras: senha com 8-20 chars, pelo menos 1 número, 1 letra maiúscula. Aplique as duas técnicas combinadas.

Script K6 progressivo com ramp-up, thresholds e análise de p95/p99

1
Teste de carga K6: ramp-up, thresholds e análise de gargalos

Crie um script K6 completo com ramp-up progressivo e thresholds de qualidade para a API JSONPlaceholder.

  • 1. Instale o K6: k6.io/docs/get-started/installation. Crie load-test.js com o script abaixo e rode: k6 run load-test.js.
  • 2. Analise o output: http_req_duration (p90, p95, p99 em ms), http_req_failed (deve ser 0%), vus ativos. O que o p99 significa para o usuário final?
  • 3. Adicione thresholds: thresholds: { "http_req_duration": ["p(95)<500"], "http_req_failed": ["rate<0.01"] }. O K6 retornará exit code 1 se os critérios falharem — útil para CI/CD.
  • 4. Aumente target em stages gradualmente (100, 200, 500 usuários). Documente em qual ponto p95 ultrapassa 1000ms — este é o limite de escala da API.
// load-test.js import http from 'k6/http'; import { sleep, check } from 'k6'; export const options = { stages: [ { duration: '30s', target: 10 }, { duration: '1m', target: 50 }, { duration: '20s', target: 0 }, ], }; export default function () { const res = http.get('https://jsonplaceholder.typicode.com/posts/1'); check(res, { 'status 200': (r) => r.status === 200, 'tempo < 500ms': (r) => r.timings.duration < 500, }); sleep(1); }
💡 Dica de início: NUNCA execute testes de carga em produção sem autorização! Use homologação ou APIs de prática. O p95 é mais útil que a média — se p95=800ms, 5% dos usuários esperam mais de 800ms a cada requisição.

Entenda a estrutura de JWT e como validar a segurança da autenticação de uma API.

  • 3. Teste de integridade: modifique 1 caractere do token e envie numa requisição autenticada. A API deve retornar 401. Documente o resultado.
  • 4. Crie um checklist de boas práticas JWT: algoritmo forte (RS256 > HS256), expiração curta, sem dados sensíveis no payload, HTTPS obrigatório.
💡 Dica de início: O payload do JWT é apenas Base64 — não está criptografado! Qualquer pessoa com o token pode ler o conteúdo. Nunca armazene senha, CPF ou dados sensíveis no payload. O que protege o JWT é a assinatura, não o encoding.
